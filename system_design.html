1. Basic agents (single user)

At the smallest scale, an agent is a loop: observe state → reason (LLM) → act (tools) → observe again. One process, one user, no persistence beyond the session.

Core loop

  • Observe – Current context: user message, conversation history, tool results, or environment state.
  • Reason – Call an LLM with a prompt (system + history + “what to do next”). Output may be a final answer or a tool call (name + arguments).
  • Act – If the model requests a tool, execute it (API, DB, code). Append the result to context and loop.
  • Terminate – Stop when the model returns a user-facing answer or when max steps/time is reached.

Minimal architecture

flowchart LR
    Client([Client]) --> Agent["Agent: observe / reason / act"]
    Agent --> LLM[LLM_API]
    Agent --> Exec[Tool_executor]
    Exec --> Tools[Tool_services]

Components: one agent service (or script), an LLM provider (OpenAI, Anthropic, local), and optional tools (HTTP, DB, filesystem). State lives in memory for the duration of the request.

When this is enough

  • Demos, internal tools, single-tenant apps.
  • Low QPS; no need for horizontal scaling or strict SLAs.

2. Multi-user & concurrency

Multiple users hit the same agent. You need stateless design, per-user isolation, and resource limits so one user cannot starve others.

Design choices

  • Stateless agent – No in-process memory of past sessions. Session state (conversation history, pending tool calls) is stored in a store (DB, Redis) keyed by session_id or user_id.
  • Concurrency – Handle many requests in parallel: async I/O (e.g. async HTTP to LLM and tools), worker pool, or event loop. Each request gets its own context and timeout.
  • Rate limiting – Per user or per API key: limit requests/minute and/or concurrent agent runs. Protects LLM quota and backend tools.
  • Connection pooling – Reuse HTTP connections to the LLM API and to DBs; avoid opening a new connection per request.

3. Reliability & production readiness

Agents call external LLMs and tools that can fail or be slow. Production systems need retries, queues, idempotency, and observability.

Concepts

  • Retries with backoff – LLM and tool calls can timeout or return 5xx. Retry with exponential backoff and jitter; set max retries and circuit-break if the dependency is down.
  • Request queue – Instead of processing every request immediately, enqueue it (e.g. Redis, SQS, Kafka). Workers pull from the queue. Smooths bursts and allows prioritization.
  • Idempotency – For critical actions (e.g. payments, writes), use idempotency keys so duplicate requests do not apply the same action twice. Store “request_id → result” and return cached result on replay.
  • Observability – Logs (request id, user, model, latency, token usage), metrics (latency p99, error rate, cost per request), and traces (full agent trace: LLM calls, tool calls, timings). Essential for debugging and cost control.

Production checklist

  • Structured logging with correlation IDs.
  • Timeouts on LLM and tool calls; graceful degradation or user-facing error.
  • Health checks for the agent and dependencies (LLM, DB, Redis).

4. Scale: thousands of users

At thousands of concurrent users you need horizontal scaling, caching, and efficient data access. The agent remains a central orchestrator, but supporting services are scaled and optimized.

Architecture elements

  • Load balancing – Distribute traffic across multiple agent instances (round-robin, least connections, or consistent hash by session_id for stickiness).
  • Caching – Cache LLM responses for identical or near-identical prompts (e.g. common system prompts, FAQ). Cache tool results where safe (e.g. read-only, short TTL). Use Redis or similar; watch invalidation and memory.
  • Database design – Sessions and conversation history: partition by user_id or session_id; index for “last N messages for session.” Consider separate read replicas for history reads. For analytics, stream events to a data warehouse.
  • Async and streaming – Stream LLM tokens to the client so the user sees progress; run tool execution in the background where possible. Use Server-Sent Events or WebSockets for real-time updates.

5. Scale: millions of users

At millions of users, the system is distributed: many regions, dedicated pipelines for different workloads, strict cost and latency budgets, and agent orchestration that can route and scale independently.

Architecture themes

  • Multi-region – Deploy agent and dependencies in several regions. Route users to the nearest region (latency-based DNS or global LB). Replicate session and config data; keep agent runs local to a region to avoid cross-region latency in the loop.
  • Async job pipeline – For non–real-time workloads (e.g. “analyze this doc,” “summarize this thread”), accept the request, enqueue a job, return a job_id. Workers process the queue; clients poll or get webhooks. Decouples user-facing latency from agent run time.
  • LLM routing & fallbacks – Route by model (e.g. cheap model for simple intents, larger model for complex). Use fallbacks (e.g. primary provider → secondary) and load shedding (reject or queue when overloaded) to protect SLAs and cost.
  • Cost control – Per-user or per-tenant limits (requests/day, tokens/month). Cache aggressively; use smaller/faster models where possible. Meter and bill by token or request; alert on anomalies.
  • Agent orchestration layer – A dedicated service that: assigns a run to a worker pool (or serverless), manages long-running or multi-turn sessions, applies policies (rate limits, model choice, tool allowlists), and records traces. Agents become workers that the orchestrator invokes.

  • Orchestrator – Receives “start/continue agent” requests; enqueues work or invokes workers; tracks session state; returns stream or job_id.
  • Worker pool – Stateless agent processes (or containers). Pull from a regional queue or get RPC from orchestrator. Scale based on queue depth and latency SLOs.
  • Data – Sessions and history in a scalable DB (sharded by user/session). Events and logs to a stream (Kafka, Kinesis) for analytics and replay. Caches (e.g. Redis cluster) per region.

State of the art at scale

  • Separate paths for real-time (sync/streaming) vs batch (async jobs).
  • Strict rate limits and quotas; cost and quality monitored per tenant.
  • Orchestrator as the single place for routing, policy, and observability.

6. Key building blocks

Regardless of scale, these components appear in most agent systems.

  • LLM gateway – Abstraction over one or more providers (OpenAI, Anthropic, etc.). Handles auth, retries, logging, and optional caching. Agent code calls “complete” or “chat” through the gateway.
  • Tool execution – Sandboxed execution of tools (HTTP, DB, code). Validate inputs; enforce timeouts and allowlists; log inputs/outputs for safety and debugging.
  • Memory / state – Short-term: conversation buffer (last K messages or tokens). Long-term: vector store or DB for “memories” or RAG. At scale, this is the session store and any user-scoped storage.
  • Auth & identity – Identify the user (or API key) for rate limits, quotas, and audit. Pass identity through the entire chain (agent, tools, logs).

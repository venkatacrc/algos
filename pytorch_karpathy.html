PyTorch syntax explained (for novices)

Short definitions of syntax that does non-obvious work. This glossary does not list every symbol in the file: basics like nn.Module, forward(), and .to(device) are explained in their sections; vision/audio and problem-specific syntax (Conv2d, MSE, etc.) appear in §18 and §19. Refer back here when you see an unfamiliar call.

Model & attention

model.train() / model.eval()
Switch training vs inference mode. train(): enables dropout, BatchNorm uses batch stats. eval(): disables dropout, BatchNorm uses running mean/var. Call model.eval() before validation or inference.
torch.no_grad()
Context manager: disables gradient tracking inside the block. Use during validation or inference to save memory and speed (no need to build the backward graph).
.to(device) / .to(device, non_blocking=True)
Moves a tensor or model to a device ("cuda", "cpu", or torch.device("cuda", index)). non_blocking=True allows async copy to GPU while CPU continues; use for data loading.
.split(size, dim) / .transpose(-2, -1) / @
split: splits a tensor along dim into chunks of size (e.g. qkv.split(n_embd, dim=2) → three tensors). transpose: swaps two dimensions (e.g. k.transpose(-2,-1) for attention). @: matrix multiply (e.g. q @ k.T = attention scores).
F.softmax(x, dim=-1)
Turns logits into probabilities: exponentiates and normalizes so they sum to 1 along dim. In attention, applied to scores so each row is a probability distribution over keys.
c_attn
In GPT-style code, the name of a single linear layer that produces Q, K, V in one go: c_attn(x) is a linear map from hidden size to 3 * hidden_size. The output is then split into three parts to get query, key, and value. “c” often means “combined” or “convolution” in older naming; here it’s just one big linear layer for efficiency.
masked_fill(mask, value)
Fills elements of the tensor where mask is True with value. In attention: att.masked_fill(..., float('-inf')) sets future positions to −∞ so that after softmax they become 0 (no attention to future tokens). That’s the “causal” mask: each position can only attend to itself and the past.

Loss & training

F.cross_entropy(logits, targets)
Computes the cross-entropy loss for classification. logits: raw scores per class, shape (N, C) or (N, T, C). targets: integer class indices (e.g. next-token IDs), shape (N,) or (N, T). Internally it applies log_softmax to logits then picks the log-prob of the target class and negates it. Returns a scalar by default (mean over all elements).
ignore_index=pad_id
In F.cross_entropy: target positions equal to pad_id are not counted in the loss (no gradient, not in the mean). Use for sequence loss when you pad to fixed length and only want to predict non-padding tokens.
optimizer.step() / scheduler.step()
optimizer.step(): updates each parameter using its .grad (e.g. SGD: param -= lr * param.grad). Call after backward(). scheduler.step(): updates the learning rate (e.g. after each batch or epoch, depending on scheduler).
clip_grad_norm_(parameters, max_norm)
Rescales all gradients so that the total norm of the gradient vector (over all parameters) is at most max_norm. If the norm is already ≤ max_norm, nothing changes. This prevents huge gradient updates that can cause training to blow up; common in transformers (e.g. max_norm=1.0). The trailing underscore means it modifies tensors in place.
zero_grad(set_to_none=True)
Clears the .grad attribute on all parameters. set_to_none=True sets param.grad = None instead of filling with zeros, which can save memory and is the recommended default in modern PyTorch.

Finetuning & SFT

sft_loss
Supervised finetuning loss: the same as next-token prediction (cross-entropy), but only on the assistant’s reply. The prompt and padding positions are masked out so the model is not trained to predict them. So: forward on full sequence, compute per-token CE with reduction="none", then multiply by response_mask and sum only over positions where the mask is 1, and divide by the number of those positions to get a scalar loss.

Autograd basics

requires_grad=True
On a tensor, enables gradient tracking: PyTorch will record every op that uses this tensor so it can compute gradients with respect to it. Parameters of nn.Module have this True by default.
loss.backward()
Fills the .grad of every tensor that took part in computing loss (and has requires_grad=True). Only call it once per forward (or use retain_graph=True in special cases). Must be called before optimizer.step() so the optimizer can use those gradients.
.grad
After backward(), each parameter’s .grad holds the gradient of the loss with respect to that parameter (same shape as the parameter). The optimizer uses these to update the weights.

Other common bits

reduction="none"
In F.cross_entropy: return a loss per element (same shape as targets) instead of averaging to a scalar. Lets you mask or weight certain positions (e.g. only assistant tokens) before summing.
.contiguous().view(...)
contiguous() makes the tensor’s memory layout contiguous (no strides that “skip” elements). view(...) reshapes without copying when the tensor is contiguous. After slicing (e.g. logits[:, :-1]) the tensor might be non-contiguous; view then can fail, so we call contiguous() first.
state_dict
A Python dict mapping parameter names (strings) to tensors. model.state_dict() gives all weights and buffers; load_state_dict(ckpt) loads them back. Used for saving/loading checkpoints. strict=True means every key in the checkpoint must match a parameter in the model (and vice versa), or load raises an error.
map_location="cpu"
When loading with torch.load(path, map_location="cpu"), tensors are moved to CPU. Useful when the checkpoint was saved on GPU but you’re loading on a machine without that GPU (or before calling .to(device)).
autocast(device_type='cuda', dtype=torch.bfloat16)
Context manager: inside the block, eligible ops (matmuls, convolutions, etc.) run in bfloat16 instead of float32. Speeds up training and reduces memory; the rest of the logic (e.g. loss scaling) is handled separately (e.g. GradScaler for float16).
scaler.scale(loss).backward() / unscale_ / step / update
With float16, gradients can underflow. The scaler multiplies the loss by a factor before backward so gradients are in a representable range; unscale_(optimizer) divides the parameter gradients by that factor before the optimizer step; step(optimizer) then updates weights; update() adjusts the scale factor for the next iteration (e.g. if no inf/nan was seen, increase scale).
log_softmax + gather (for log probs)
To get the log probability of the actual next token: apply log_softmax to logits (so each position has a distribution over vocab). Then gather(dim, index) picks the log-prob at the index of the true token. Sum over positions (with mask) gives the sequence log probability used in RL (e.g. PPO/DPO).

1. Autograd & backprop

PyTorch’s autograd records ops on tensors and computes gradients via backprop. Understanding this is easier if you see a minimal version first.

PyTorch syntax & usage

import torch

# Tensors track ops when requires_grad=True
x = torch.tensor([2.0], requires_grad=True)
y = x ** 2 + 3 * x
y.backward()          # compute gradients
print(x.grad)         # dy/dx = 2*x + 3 → 7.0

# In training: loss is a scalar; backward() fills .grad on all parameters
loss = model(inputs).mean()
loss.backward()       # populates parameter.grad
optimizer.step()      # optimizer uses .grad to update parameters

Example: input and output

Input: x = tensor([2.0], requires_grad=True). Expression: y = x**2 + 3*x.

Forward: y = 4 + 6 = 10.0. Derivative dy/dx = 2x + 3 → at x=2: 2*2 + 3 = 7.

Output after y.backward(): x.grad is tensor([7.0]).

Where to see it

  • micrograd/engine.py – Value class and backward() (~100 lines). Builds intuition for loss.backward().
  • minGPT mingpt/trainer.py – loss.backward(), gradient clipping.
  • nanoGPT train.py – loss.backward(), gradient accumulation, scaler for mixed precision.

2. Tensors, nn.Module, parameters

Models in PyTorch are nn.Modules; parameters are nn.Parameter and get gradients. Submodules and parameters are registered automatically.

PyTorch syntax & usage

import torch
import torch.nn as nn

class MyModel(nn.Module):
    def __init__(self, in_dim, out_dim):
        super().__init__()
        self.linear = nn.Linear(in_dim, out_dim)   # registered submodule
        self.scale = nn.Parameter(torch.ones(1))  # registered parameter

    def forward(self, x):
        return self.scale * self.linear(x)

model = MyModel(64, 10)
list(model.parameters())   # all parameters (from linear + scale)
model(x)                   # calls forward(x)

Example: input and output

Input: x of shape (batch_size, 64), e.g. x = torch.randn(4, 64).

Output: model(x) has shape (4, 10). list(model.parameters()) returns 3 tensors: linear weight (10, 64), linear bias (10,), and scale (1,).

Where to see it

  • minGPT mingpt/model.py – GPT as nn.Module, config-driven layers.
  • nanoGPT model.py – GPTConfig, GPT nn.Module, embed, blocks, lm_head.
  • nanochat – Model defs and parameter groups: scripts/base_train.py.

3. Transformers: attention, blocks, GPT

Multi-head self-attention, layer norm, MLP blocks, causal masking—core of GPT-style models.

PyTorch syntax & usage

import torch
import torch.nn as nn
import torch.nn.functional as F

# Causal self-attention: Q,K,V from one linear, then mask future positions
B, T, C = batch, seq_len, n_embd
qkv = self.c_attn(x)  # (B,T,3*C)
q, k, v = qkv.split(n_embd, dim=2)
att = (q @ k.transpose(-2, -1)) * (1.0 / (C ** 0.5))
att = att.masked_fill(self.bias[:, :, :T, :T] == 0, float('-inf'))
att = F.softmax(att, dim=-1)
out = att @ v

# Pre-norm block: LayerNorm -> Attention -> residual; LayerNorm -> MLP -> residual
x = x + self.attn(self.ln1(x))
x = x + self.mlp(self.ln2(x))

Example: shapes

Input: x shape (B, T, C) e.g. (2, 128, 768) (batch=2, seq_len=128, n_embd=768).

After c_attn(x): qkv is (2, 128, 3*768); split gives q, k, v each (2, 128, 768). att = q @ k.T is (2, 128, 128); causal mask sets future positions to -inf before softmax. Output: out same shape as v: (2, 128, 768).

Where to see it

  • minGPT mingpt/model.py – Block, CausalSelfAttention, MLP.
  • nanoGPT model.py – CausalSelfAttention, Block; single matmul for Q,K,V.
  • nanochat – Transformer stack: model.py (or equivalent in repo).

4. Training loop: loss, backward, optimizer

Standard pattern: forward → loss → backward → optimizer step (with optional gradient accumulation and clipping).

PyTorch syntax & usage

model.train()
for batch in dataloader:
    x, y = batch
    logits = model(x)
    loss = F.cross_entropy(logits, y)

    optimizer.zero_grad(set_to_none=True)  # clear previous gradients
    loss.backward()                         # compute gradients
    torch.nn.utils.clip_grad_norm_(model.parameters(), max_norm=1.0)
    optimizer.step()
    scheduler.step()                        # optional: LR schedule

Example: one step

Input: batch = (x, y) with x shape (32, 128) (batch 32, seq 128), y shape (32, 128) (target next-token ids).

Forward: logits = model(x) → shape (32, 128, vocab_size). loss = F.cross_entropy(logits.view(-1,V), y.view(-1)) → scalar, e.g. 3.45.

After loss.backward(): every param.grad is filled. optimizer.step() updates params using those grads; next iteration zero_grad clears them.

# Gradient accumulation (effective batch = batch_size * accum_steps)
optimizer.zero_grad(set_to_none=True)
for i, batch in enumerate(dataloader):
    loss = model(batch) / accum_steps
    loss.backward()
    if (i + 1) % accum_steps == 0:
        optimizer.step()
        optimizer.zero_grad(set_to_none=True)

Where to see it

  • minGPT mingpt/trainer.py – Training loop: batch, model(x), loss.backward(), optimizer.step(), scheduler.
  • nanoGPT train.py – Main loop, gradient accumulation, scaler, logging.
  • nanochat scripts/base_train.py – Pretrain/SFT loops, multi-stage training.

5. Optimizers (AdamW, Muon)

AdamW is the default for LLMs; nanochat adds Muon for main weights with Adam for embeddings.

PyTorch syntax & usage

import torch.optim as optim

# AdamW: decoupled weight decay, standard for transformers
optimizer = optim.AdamW(
    model.parameters(),
    lr=3e-4,
    weight_decay=0.01,
    betas=(0.9, 0.95)
)

# Learning rate scheduler (e.g. cosine decay with warmup)
from torch.optim.lr_scheduler import CosineAnnealingLR, LinearLR, SequentialLR
warmup = LinearLR(optimizer, start_factor=0.1, total_iters=warmup_iters)
decay = CosineAnnealingLR(optimizer, T_max=total_iters)
scheduler = SequentialLR(optimizer, [warmup, decay], [warmup_iters])

Example: optimizer and scheduler

Input: model.parameters() (e.g. 124M params), lr=3e-4, weight_decay=0.01.

After first optimizer.step(): params updated; param.grad still present until zero_grad(). With LinearLR(start_factor=0.1, total_iters=100), step 0 gives effective lr 3e-5; step 100 gives 3e-4; then cosine decay reduces it toward 0.

Where to see it

  • nanoGPT train.py – AdamW, weight decay, LR; scheduler.
  • minGPT mingpt/trainer.py – AdamW config.
  • nanochat scripts/base_train.py – Muon + Adam parameter groups (search for optimizer).

6. Distributed & multi-GPU (DDP, FSDP, torchrun)

Data-parallel training across GPUs/nodes: DDP replicates the model and syncs gradients each step; FSDP (Fully Sharded Data Parallel) shards parameters, gradients, and optimizer state across ranks so very large models can fit. Launchers like torchrun start one process per GPU.

Concepts

  • DDP – Each rank holds a full copy of the model. After backward, gradients are all-reduced so all replicas stay in sync. Good when the model fits in one GPU’s memory.
  • FSDP – Parameters, gradients, and optimizer states are sharded across ranks; each rank only keeps 1/world_size. During forward/backward, all-gather brings the full layer for that step, then you free the gathered copy. Reduces per-GPU memory so you can scale to larger models or larger batch sizes.
  • Process group – init_process_group(backend="nccl") ties the processes together; LOCAL_RANK / RANK / WORLD_SIZE from the launcher.

PyTorch syntax & usage (DDP)

import torch.distributed as dist
from torch.nn.parallel import DistributedDataParallel as DDP

# Initialize process group (each process runs this)
dist.init_process_group(backend="nccl")  # or "gloo" for CPU
local_rank = int(os.environ["LOCAL_RANK"])
torch.cuda.set_device(local_rank)
device = torch.device("cuda", local_rank)

model = MyModel().to(device)
model = DDP(model, device_ids=[local_rank])

# Launch with: torchrun --nproc_per_node=8 train.py

PyTorch syntax & usage (FSDP)

import torch.distributed as dist
from torch.distributed.fsdp import FullyShardedDataParallel as FSDP, StateDictType
from torch.distributed.fsdp import ShardingStrategy, MixedPrecision

# After init_process_group, wrap model with FSDP
# ShardingStrategy: FULL_SHARD (default, shard params/grads/optim state), SHARD_GRAD_OP, or NO_SHARD
model = MyModel()
model = FSDP(
    model,
    sharding_strategy=ShardingStrategy.FULL_SHARD,
    device_id=local_rank,
    mixed_precision=MixedPrecision(
        param_dtype=torch.bfloat16,
        reduce_dtype=torch.bfloat16,
    ),
)
# Optimizer sees flattened parameter groups from FSDP
optimizer = torch.optim.AdamW(model.parameters(), lr=1e-4)

# Save/load: use state_dict_type and rank 0 for full state, or distributed checkpoint
with FSDP.state_dict_type(model, StateDictType.FULL_STATE_DICT):
    state = model.state_dict()
    if dist.get_rank() == 0:
        torch.save(state, "model.pt")

Where to see it

  • nanoGPT train.py – DDP init, DistributedDataParallel; config/.
  • nanochat speedrun.sh – torchrun launch; scripts/base_train.py for dist setup.
  • PyTorch FSDP – torch.distributed.fsdp; FSDP tutorial.

7. Distributed data loader & optimization

In multi-GPU training, each rank must see a different subset of the data (no duplicate batches). Use DistributedSampler so each rank gets non-overlapping indices. Then optimize the loader with memory (pinned memory, persistent workers) and cache layers (in-memory or memory-mapped data) so the GPU is not waiting on I/O.

Concepts

  • DistributedSampler – Partitions dataset indices across ranks; each rank gets len(dataset) // world_size samples per epoch. Call sampler.set_epoch(epoch) so shuffling differs each epoch.
  • Memory optimization – pin_memory=True uses pinned (page-locked) host memory so transfer to GPU is async and faster. persistent_workers=True keeps worker processes alive between epochs (avoids fork overhead). prefetch_factor controls how many batches each worker prefetches.
  • Cache layers – Load the full dataset (or a shard) into RAM once and serve from memory; or use memory-mapped arrays (np.memmap, torch.from_file) so the OS pages in data on access. Reduces repeated disk reads and speeds up epoch start.

PyTorch syntax & usage

from torch.utils.data import DataLoader, DistributedSampler

# Distributed sampler: each rank gets a unique subset
sampler = DistributedSampler(
    dataset,
    num_replicas=dist.get_world_size(),
    rank=dist.get_rank(),
    shuffle=True,
)
dataloader = DataLoader(
    dataset,
    batch_size=per_device_batch,
    sampler=sampler,  # do not pass shuffle=True when using sampler
    num_workers=4,
    pin_memory=True,
    persistent_workers=True,
    prefetch_factor=2,
)

# In training loop, set epoch for correct shuffling
for epoch in range(epochs):
    sampler.set_epoch(epoch)
    for batch in dataloader:
        batch = {k: v.to(device, non_blocking=True) for k, v in batch.items()}
        ...

Example: input and output

Input: dataset with 80_000 samples, world_size=4, batch_size=32. DistributedSampler gives rank 0 indices 0,4,8,...; rank 1 indices 1,5,9,...; etc. (20_000 per rank).

Output: Each rank’s dataloader yields 20_000/32 ≈ 625 batches per epoch. set_epoch(epoch) changes the shuffle seed so each epoch sees a different order. batch.to(device, non_blocking=True) keeps transfer async with compute.

# Cache layer: load dataset into RAM once (if it fits)
class CachedDataset(torch.utils.data.Dataset):
    def __init__(self, underlying, load_all=True):
        self.underlying = underlying
        self.cache = []
        if load_all:
            for i in range(len(underlying)):
                self.cache.append(underlying[i])

    def __len__(self):
        return len(self.underlying)

    def __getitem__(self, i):
        if self.cache:
            return self.cache[i]
        return self.underlying[i]

# Or memory-mapped: dataset lives on disk, OS pages it in
import numpy as np
data = np.memmap("train.bin", dtype=np.uint16, mode="r", shape=(N, block_size))
# In Dataset.__getitem__, slice data[i] and convert to tensor; no full load

8. Mixed precision & FP8

FP16/BF16 mixed precision with GradScaler; FP8 on H100+ for memory and speed.

PyTorch syntax & usage

from torch.amp import autocast, GradScaler

scaler = GradScaler('cuda')  # for float16; bfloat16 often doesn't need scaling

for batch in dataloader:
    optimizer.zero_grad(set_to_none=True)
    with autocast(device_type='cuda', dtype=torch.bfloat16):  # or torch.float16
        logits = model(batch)
        loss = F.cross_entropy(logits, targets)
    scaler.scale(loss).backward()
    scaler.unscale_(optimizer)
    torch.nn.utils.clip_grad_norm_(model.parameters(), 1.0)
    scaler.step(optimizer)
    scaler.update()

Example: input and output

Input: batch (e.g. input_ids on GPU), model in float32. Inside autocast(dtype=torch.bfloat16), linear/matmul ops run in bfloat16; logits and loss may be bfloat16.

Output: scaler.scale(loss) scales loss (e.g. 2^12) so backward produces finite gradients; unscale_ undoes scale before step(); scaler.update() adjusts scale for next iteration. Effective behavior: same training as FP32 with lower memory and often higher throughput.

Where to see it

  • nanoGPT train.py – autocast, bfloat16/float16, GradScaler.
  • nanochat scripts/base_train.py – FP8 / autocast (search for amp or dtype).

10. Data loading & tokenization

Datasets, tokenizers (BPE, tiktoken), batching, and DataLoader.

PyTorch syntax & usage

from torch.utils.data import Dataset, DataLoader

class TokenDataset(Dataset):
    def __init__(self, tokens, block_size):
        self.data = torch.tensor(tokens, dtype=torch.long)
        self.block_size = block_size
    def __len__(self):
        return len(self.data) - self.block_size
    def __getitem__(self, i):
        x = self.data[i : i + self.block_size]
        y = self.data[i + 1 : i + self.block_size + 1]
        return x, y

dataset = TokenDataset(encoded, block_size=128)
dataloader = DataLoader(dataset, batch_size=32, shuffle=True, num_workers=4)
# Tokenization: tiktoken (OpenAI) or Hugging Face
import tiktoken
enc = tiktoken.get_encoding("gpt2")
tokens = enc.encode("Hello world")
text = enc.decode(tokens)

Example: input and output

TokenDataset: If encoded has length 10_000 and block_size=128, len(dataset) is 9872. dataset[i] returns (x, y) where x and y are 1D tensors of length 128; y is x shifted by one position.

DataLoader: One batch: x shape (32, 128), y shape (32, 128) (batch_size=32).

Tiktoken: enc.encode("Hello world") → e.g. [15496, 995] (GPT-2). enc.decode([15496, 995]) → "Hello world".

11. Data preparation

Before training, raw text or conversation data is turned into token IDs, chunked into fixed-length blocks, and split into train/val. For pretraining you often train a BPE tokenizer on the same corpus; for SFT you format instruction–response pairs and optionally pack multiple examples per sequence.

Pipeline (concepts)

  • Tokenizer – Train BPE (or use pretrained tiktoken/GPT-2). Vocabulary size and merge rules affect coverage and sequence length.
  • Chunking – Concatenate documents, then split into non-overlapping blocks of block_size tokens (e.g. 1024). No document should span a chunk boundary if you want clean boundaries, or use sliding window with overlap for more data.
  • Train/val split – Hold out a fraction (e.g. 5%) by time or by document; never mix val into train.
  • Deduplication – Optional: remove near-duplicate documents (e.g. minhash) to reduce memorization and improve generalization.
  • SFT/chat format – Each example: system + user + assistant messages. Tokenize with special tokens (e.g. <|im_start|>, <|im_end|>) and pack or pad to max length; mask loss on padding and sometimes on user turns (train only on assistant).

PyTorch / script pattern

# Pretrain data prep: one .bin file of token ids (uint16), one .meta with vocab size etc.
import numpy as np
import tiktoken
enc = tiktoken.get_encoding("gpt2")
with open("input.txt") as f:
    text = f.read()
tokens = enc.encode_ordinary(text)
block_size = 1024
# Chunk into blocks (drop last incomplete block)
data = np.array(tokens[: len(tokens) // block_size * block_size], dtype=np.uint16)
data = data.reshape(-1, block_size)
np.save("train.npy", data[:int(0.95*len(data))])

12. Finetuning models

Finetuning starts from a pretrained checkpoint and updates the model on a smaller, often task-specific dataset. Supervised finetuning (SFT) uses next-token (cross-entropy) loss on instruction–response or chat data. You can finetune all parameters or use parameter-efficient methods (LoRA, adapters).

Concepts

  • Load checkpoint – Restore state_dict from pretrained (e.g. OpenAI GPT-2 or your own pretrain run). Match model config (n_layer, n_head, etc.) or resize embeddings if vocab changed.
  • Instruction / chat format – Examples are (system, user, assistant) or (prompt, completion). Tokenize with role tokens; often only the assistant part gets loss (mask user/system in the loss).
  • Learning rate – Typically 1e-5 to 5e-5 for full finetune; 1e-4 to 3e-4 for LoRA. Lower than pretrain to avoid forgetting.
  • LoRA (Low-Rank Adaptation) – Train small low-rank matrices that multiply the pretrained weights; only LoRA params are updated. Saves memory and speeds training; merge back for inference or keep separate.

PyTorch syntax & usage

# Load pretrained checkpoint (e.g. GPT-2 124M)
from transformers import GPT2LMHeadModel
model = GPT2LMHeadModel.from_pretrained("gpt2")
# Or from local .pt: model.load_state_dict(torch.load("ckpt.pt"))

# SFT: forward on (prompt + response), loss only on response tokens
def sft_loss(model, input_ids, response_mask):
    logits = model(input_ids).logits
    shift_logits = logits[:, :-1].contiguous().view(-1, logits.size(-1))
    shift_labels = input_ids[:, 1:].contiguous().view(-1)
    loss = F.cross_entropy(shift_logits, shift_labels, reduction="none")
    loss = (loss.view(-1) * response_mask[:, 1:].reshape(-1)).sum() / response_mask[:, 1:].sum().clamp(min=1)
    return loss

# Train for a few epochs
optimizer = torch.optim.AdamW(model.parameters(), lr=2e-5)
for batch in dataloader:
    loss = sft_loss(model, batch["input_ids"], batch["response_mask"])
    loss.backward()
    optimizer.step()
    optimizer.zero_grad(set_to_none=True)

Example: SFT loss

Input: input_ids shape (4, 256) (batch 4, seq 256); response_mask shape (4, 256), 1 on assistant tokens, 0 on prompt/padding.

Forward: logits (4, 256, V). Shifted logits/labels: only positions where response_mask[:, 1:] == 1 contribute to loss. Output: scalar loss, e.g. 2.1; loss.backward() fills grads on model parameters.

Where to see it

  • nanoGPT train.py – Finetuning: load from init_from (pretrain or resume); same training loop, smaller dataset.
  • nanochat – Full pipeline: pretrain then SFT scripts (instruction/chat data, loss masking).

13. Model evaluation

Evaluation measures how well the model has learned. During training you monitor validation loss (and optionally perplexity). For downstream quality you run benchmarks (e.g. accuracy on held-out tasks) or human eval.

Concepts

  • Validation loss – Forward on val set, no backward; average cross-entropy. Lower is better. Use for early stopping and hyperparameter choice.
  • Perplexity – exp(mean cross-entropy). Interpretable as “effective branching factor”; lower is better.
  • Bits per byte (bpb) – For raw bytes: (loss in nats) / ln(2) ≈ bits per character. Vocabulary-independent, good for comparing tokenizers.
  • Benchmarks – Run model on fixed prompts (e.g. few-shot or zero-shot), parse answers, compare to gold. Examples: MMLU, GSM8K, HumanEval (code), DCLM CORE (nanochat leaderboard).

PyTorch syntax & usage

@torch.no_grad()
def evaluate(model, dataloader):
    model.eval()
    total_loss, total_tokens = 0.0, 0
    for batch in dataloader:
        input_ids = batch["input_ids"].to(model.device)
        logits = model(input_ids).logits
        loss = F.cross_entropy(
            logits[:, :-1].reshape(-1, logits.size(-1)),
            input_ids[:, 1:].reshape(-1),
            ignore_index=pad_id,
        )
        n = (input_ids != pad_id).sum().item()
        total_loss += loss.item() * n
        total_tokens += n
    return total_loss / total_tokens  # mean loss

# Perplexity
import math
val_loss = evaluate(model, val_loader)
ppl = math.exp(val_loss)
# Bits per byte (if tokenizer is byte-level or you measure per byte)
bpb = val_loss / math.log(2)

Example: input and output

Input: model, val_loader yielding batches of input_ids (e.g. shape (32, 128)), pad_id=0.

Output: val_loss = mean cross-entropy over non-padding positions, e.g. 3.2. ppl = exp(3.2) ≈ 24.5. bpb (bits per byte) ≈ 3.2 / 0.693 ≈ 4.6 when measuring per-byte.

  • nanochat – Evaluation and DCLM CORE / leaderboard; scripts for running benchmarks.
  • autoresearch – prepare.py: val bits-per-byte (val_bpb) as the metric the agent optimizes.

14. Checkpoints, config, and inference

Checkpointing saves model and optimizer state so you can resume training or load for inference. Config (YAML, dataclass, or CLI) keeps hyperparameters in one place. Inference runs the model without gradients, often with batching and KV-cache for generation.

Concepts

  • Save – torch.save({"model": model.state_dict(), "optimizer": optimizer.state_dict(), "step": step}, path). Optionally save config and RNG state for exact resume.
  • Load – model.load_state_dict(ckpt["model"]); match keys and shapes (strict=True). For inference, only model state is needed.
  • Config – Single source of truth for batch size, LR, block size, n_layer, etc. nanoGPT uses a config dict / YAML; override via CLI (e.g. --batch_size 32).
  • Inference – model.eval(); torch.no_grad(); model.generate(...). KV-cache caches key/value for previous tokens to avoid recomputing; use for long generations.

PyTorch syntax & usage

# Save checkpoint (training)
ckpt = {"model": model.state_dict(), "optimizer": optimizer.state_dict(), "iter_num": iter_num, "best_val_loss": best_val_loss}

# Load for resume or inference

# Inference: no grad, generate with optional KV-cache

15. Reinforcement learning (RL) post-training loop

After pretraining and SFT, many LLM pipelines add an RL phase: the policy (current model) is updated to maximize a reward (e.g. from a reward model, or from preference data) while staying close to a reference policy via a KL penalty. The “inner” loop is: sample or load responses → get log probs from policy and reference → compute reward/advantage → form loss → backward.

Inner workings (concepts)

  • Policy π_θ – The model being trained; outputs logits over next tokens.
  • Reference policy π_ref – Frozen copy of the model before RL; used to compute KL(π_θ ‖ π_ref) so the policy doesn’t drift too far.
  • Reward – Scalar per response (e.g. from a reward model, or from preference data). Often normalized to advantage (reward minus baseline).
  • Log probs – For each token in the response, log π(a_t | s_t). Sum over the response gives log π(response | prompt).
  • PPO-style – Ratio r_t = π_θ(a_t|s_t) / π_ref(a_t|s_t). Loss uses clipped ratio × advantage plus KL penalty.
  • DPO-style – No reward model. Preference pairs (y_w, y_l). Loss from log σ(β log(π_θ(y_w)/π_ref(y_w)) − β log(π_θ(y_l)/π_ref(y_l))).

PyTorch syntax & usage (policy log probs, ref log probs, loss)

# Get log probs for a batch of (prompt, response) sequences
def get_log_probs(model, input_ids, attention_mask):
    logits = model(input_ids).logits
    # logits: (B, T, V). Shift to get logits for next token at each position
    log_probs = F.log_softmax(logits[:, :-1], dim=-1)
    # Gather log prob of the actual next token
    next_token_ids = input_ids[:, 1:]
    token_log_probs = log_probs.gather(-1, next_token_ids.unsqueeze(-1)).squeeze(-1)
    # Mask padding; sum over non-padding positions
    mask = attention_mask[:, 1:].float()
    return (token_log_probs * mask).sum(-1) / mask.sum(-1).clamp(min=1e-8)

# RL step: policy vs reference, then loss
policy_logps = get_log_probs(policy_model, input_ids, attention_mask)
ref_logps = get_log_probs(ref_model, input_ids, attention_mask)  # no grad
ratio = torch.exp(policy_logps - ref_logps)
kl_penalty = (ref_logps - policy_logps).mean()
# PPO-style clipped objective (advantage A from reward - baseline)
advantage = rewards - baseline
surr1 = ratio * advantage
surr2 = torch.clamp(ratio, 0.8, 1.2) * advantage
loss = -torch.min(surr1, surr2).mean() + beta * kl_penalty
loss.backward()
optimizer.step()

Example: log probs and PPO loss

Input: input_ids (B, T) e.g. (8, 64), attention_mask (8, 64). get_log_probs(policy_model, ...) returns shape (8,): mean log prob per sequence. rewards shape (8,), baseline scalar.

Output: ratio = exp(policy_logps - ref_logps) shape (8,); advantage = rewards - baseline. loss is a scalar; after backward(), only policy_model parameters get gradients (ref_model is frozen).

# DPO-style: preference pairs (y_w preferred over y_l), no reward model
# Concatenate prompt+response for chosen and rejected
logps_chosen = get_log_probs(policy_model, chosen_ids, chosen_mask)
logps_rejected = get_log_probs(policy_model, rejected_ids, rejected_mask)
ref_logps_chosen = get_log_probs(ref_model, chosen_ids, chosen_mask)
ref_logps_rejected = get_log_probs(ref_model, rejected_ids, rejected_mask)
# DPO loss: -log sigma(beta * (log_ratio_chosen - log_ratio_rejected))
log_ratio_chosen = logps_chosen - ref_logps_chosen
log_ratio_rejected = logps_rejected - ref_logps_rejected
loss = -F.logsigmoid(beta * (log_ratio_chosen - log_ratio_rejected)).mean()
loss.backward()
optimizer.step()

Where to see it

  • trl – Hugging Face TRL: PPOTrainer, DPOTrainer; full PPO/DPO loops and reward handling.

16. AI agent inner workings

An AI agent is a loop that repeatedly observes state, reasons (optionally with an LLM), takes an action (e.g. edit code, call a tool), and observes the outcome. Training runs or tool results become the next observation. The “inner” part is the control flow, how you call the model, and how you parse and execute actions.

Inner workings (concepts)

  • Observe – Current state: task description, conversation history, last tool output, file contents, or metrics (e.g. val loss).
  • Reason / plan – LLM generates next step (natural language, or structured tool call). Can be one prompt per turn or multi-step (ReAct: Thought → Action → Observation).
  • Act – Execute the action: run code, call an API, edit a file, run training for N minutes. Actions are often constrained by a schema or a fixed set of tools.
  • Memory / context – Append only new content (e.g. tool result) to keep context under the model’s length limit; optionally summarize or truncate old turns.
  • Termination – Stop when the model outputs “done”, or when a goal is met (e.g. metric improved), or when max steps are reached.

PyTorch / inference in the loop

The agent loop usually uses the model in inference (no backward). You build a prompt from state, call model.generate(), parse the output for tool calls or edits, run them, then feed the result back into the next prompt.

# Minimal agent loop: prompt → generate → parse → act → append result → repeat
def agent_loop(model, tokenizer, system_prompt, tools, max_steps=10):
    messages = [{"role": "system", "content": system_prompt}]
    for step in range(max_steps):
        prompt = format_messages(messages)
        input_ids = tokenizer(prompt, return_tensors="pt").input_ids.to(model.device)
        with torch.no_grad():

                input_ids,

        response = tokenizer.decode(out_ids[0][input_ids.shape[1]:], skip_special_tokens=True)
        messages.append({"role": "assistant", "content": response})

        if is_done(response):
            break
        tool_name, tool_args = parse_tool_call(response)
        result = tools[tool_name](**tool_args)
        messages.append({"role": "user", "content": f"Tool result: {result}"})
    return messages

17. Low-level CUDA and C++ optimizations

Training and inference speed often come from CUDA kernels (custom or library), kernel fusion, efficient matmuls, and memory layout. PyTorch uses cuDNN/cuBLAS under the hood; you can add custom ops or drop to C/CUDA (e.g. llm.c) for full control.

Concepts and techniques

  • Matmul and cuBLAS/cuBLASLt – Most FLOPs are in matrix multiplies. cuBLASLt offers flexible matmul + fused bias/activation (e.g. matmul + bias + GELU in one pass). TF32 on Ampere+ gives near-FP32 precision with higher throughput.
  • Kernel fusion – Combine several small kernels (e.g. softmax + mask + scale, or linear + GELU) into one to reduce global memory traffic and launch overhead. Flash Attention fuses attention score → softmax → × value and uses tiling to cut memory.
  • CUDA graphs – Record a sequence of kernel launches and replay them with one CPU call. Cuts launch latency and improves GPU utilization; useful for fixed-shape training steps or inference.
  • Custom CUDA kernels – Loop unrolling, coalesced memory access, warp-level primitives, and shared memory for hand-tuned ops (e.g. backward bias, softmax). Target high % of peak compute or memory bandwidth.
  • Memory optimization – Activation checkpointing (recompute activations in backward to save memory); pinned host memory and async copies for data loading; avoiding unnecessary syncs.
  • C++ / CUDA extensions in PyTorch – torch.utils.cpp_extension or load_inline to compile and load custom kernels; JIT or AOT.

Where these show up (PyTorch vs raw C/CUDA)

// Raw C/CUDA (llm.c style): custom kernel for matmul + bias backward
// Fused: one kernel does bias gradient from output gradient (no separate pass)
__global__ void matmul_backward_bias_kernel(const float* dout, float* dbias, int M, int N) {
    // Coalesced reads, warp reductions, minimal casts for 7–8% speedup
    int row = blockIdx.x * blockDim.x + threadIdx.x;
    if (row < N) {
        float sum = 0.0f;
        for (int i = threadIdx.x; i < M; i += blockDim.x)
            sum += dout[i * N + row];
        dbias[row] = sum;
    }
}
// cuBLASLt: matmul + fused bias + GELU in one call (llm.c PR ~2x perf)

18. Vision, audio, multimodal & other models

PyTorch syntax commonly used for vision (CNNs, ViT), audio, multimodal (vision+language, etc.), and basic/advanced ML (MLPs, RNNs, embeddings). The Karpathy repos in this tutorial focus on LLMs; the building blocks below appear in torchvision, torchaudio, and multimodal codebases.

Vision: convolutions, pooling, BatchNorm, pretrained

Core layers

  • Conv2d(in_c, out_c, kernel_size, stride, padding) – 2D convolution over images. Input shape (B, C, H, W); output (B, out_c, H', W'). kernel_size=3, padding=1 keeps spatial size when stride=1.
  • MaxPool2d(kernel_size, stride) – Downsamples by taking max in each window. Often kernel_size=2, stride=2 → half H and W.
  • BatchNorm2d(num_features) – Normalizes each channel across the batch (zero mean, unit var); stabilizes training. Use model.eval() at inference so it uses running stats.
  • AdaptiveAvgPool2d(output_size) – Pools to a fixed size (e.g. (1, 1)) so you can flatten to a vector regardless of input resolution.
import torch.nn as nn

# Simple CNN: conv -> relu -> pool -> flatten -> linear
class SmallCNN(nn.Module):
    def __init__(self):
        super().__init__()
        self.conv1 = nn.Conv2d(3, 32, 3, padding=1)   # (B,3,H,W) -> (B,32,H,W)
        self.pool = nn.MaxPool2d(2, 2)               # -> (B,32,H/2,W/2)
        self.conv2 = nn.Conv2d(32, 64, 3, padding=1)
        self.bn = nn.BatchNorm2d(64)
        self.adapt = nn.AdaptiveAvgPool2d((1, 1))   # -> (B,64,1,1)
        self.fc = nn.Linear(64, num_classes)

    def forward(self, x):
        x = self.pool(nn.functional.relu(self.conv1(x)))
        x = self.adapt(nn.functional.relu(self.bn(self.conv2(x))))
        return self.fc(x.flatten(1))

Multimodal: two encoders, projection, fusion

Combine vision and text (or audio and text): encode each modality, project to a shared dimension, then fuse (concat, add, or cross-attention).

# Minimal vision–language: image encoder -> vector; text encoder -> vector; fuse
class VisionLanguageEncoder(nn.Module):
    def __init__(self, vision_dim=512, text_dim=768, hidden_dim=256):
        super().__init__()
        self.vision_encoder = resnet18(weights=...)
        self.vision_encoder.fc = nn.Linear(512, hidden_dim)
        self.text_proj = nn.Linear(text_dim, hidden_dim)  # e.g. from transformer last hidden

    def forward(self, image, text_embeds):
        img_feat = self.vision_encoder(image)           # (B, hidden_dim)
        text_feat = self.text_proj(text_embeds)         # (B, T, hidden_dim) or (B, hidden_dim)
        # Fusion: concat and classify, or cosine similarity for retrieval
        return img_feat, text_feat

Basic ML: Sequential, Dropout, activations, losses

  • nn.Sequential – Stacks layers; forward is apply each in order. Good for simple feedforward stacks.
  • nn.Dropout(p) – Randomly zeros elements with probability p (during train); scaled by 1/(1-p) so expectation unchanged. Disabled in eval().
  • Activations – F.relu, F.gelu, F.silu (Swish), F.sigmoid (0–1), F.softmax(dim=-1) for probabilities.
  • Losses – F.mse_loss(pred, target) regression; F.binary_cross_entropy_with_logits(pred, target) binary classification (pred = logits, no sigmoid); F.cross_entropy multi-class (see glossary).
mlp = nn.Sequential(

    nn.Dropout(0.2),
    nn.Linear(256, 10),
)
loss = F.mse_loss(pred, target)   # regression
loss = F.binary_cross_entropy_with_logits(logits, target.float())  # binary

Embeddings & RNNs

  • nn.Embedding(num_embeddings, embedding_dim) – Lookup table: integer indices → vectors. Input shape (B, T) of long indices; output (B, T, embedding_dim). Used for token/word IDs.
  • nn.LSTM / nn.GRU – Recurrent layers. Input (T, B, C) or (B, T, C) with batch_first=True; output hidden state and optionally cell (LSTM). Good for sequential data before transformers dominated.
embed = nn.Embedding(vocab_size, 256)

LayerNorm, residual, ViT-style patches

  • nn.LayerNorm(normalized_shape) – Normalizes over last dimension(s). For (B, T, C) use LayerNorm(C). Often used in transformer blocks (pre-norm).
  • Residual – x = x + sublayer(x) or x = x + sublayer(norm(x)). Helps gradient flow in deep nets.
  • ViT (Vision Transformer) – Split image into patches; flatten each patch to a vector; add position embeddings; feed to standard transformer. nn.Conv2d(in_c, embed_dim, kernel_size=patch_size, stride=patch_size) can create patch embeddings.

19. Model approaches: modeling and evaluating different problems

Different problems need different outputs, losses, and evaluation metrics. This section maps problem types to modeling choices and how to evaluate them.

Problem Model output Loss Evaluation
Multi-class classification (one label per example) Logits (B, C) or logits per token (B, T, C) F.cross_entropy(logits, targets) Accuracy, top-k accuracy; precision/recall/F1 per class; confusion matrix; ROC-AUC (if binary or one-vs-rest).
Multi-label classification (multiple labels per example) Logits (B, C); each position is independent binary F.binary_cross_entropy_with_logits(logits, targets.float()) with targets (B, C) in {0,1}. Subset accuracy; macro/micro F1; precision@k; average precision per label.
Regression Continuous (B, out_dim) F.mse_loss(pred, target) or F.l1_loss (MAE); Huber for robustness. MSE, RMSE, MAE; R²; MAPE if scale matters.
Sequence / language modeling (next-token prediction) Logits (B, T, V); predict next token at each position F.cross_entropy(logits.view(-1,V), labels.view(-1), ignore_index=pad_id) Perplexity (exp(mean CE)); bits per byte; downstream: accuracy on benchmarks (MMLU, GSM8K, etc.), human eval.
Retrieval / embedding similarity Embeddings (B, D); similarity = dot product or cosine Contrastive: InfoNCE / cross-entropy over batch (positive pair, negatives = rest of batch); triplet loss. Recall@k, MRR, NDCG; accuracy when retrieval is used for classification (e.g. k-NN).
Ranking / learning to rank Score per item (B, n_items) or per (query, doc) pair Pairwise: BCE on (score_i − score_j) vs label; listwise: ListNet, LambdaRank; pointwise: MSE/CE on relevance. NDCG, MAP, MRR; precision/recall at k.
Reinforcement learning (policy learning) Logits over actions or log prob of chosen action; value head optional Policy gradient: −log_prob * advantage; PPO: clipped objective + KL; DPO: preference loss (no reward model). Average return, success rate; for LLM RL: reward model score, preference win rate, downstream task accuracy.
Segmentation (pixel-wise label) Logits (B, C, H, W) per pixel class F.cross_entropy over spatial dims (reshape to (B*H*W, C) and labels (B*H*W)); or dice / focal for imbalance. mIoU (mean intersection over union), pixel accuracy, Dice; per-class IoU.
Object detection Boxes (bbox) + class logits per proposal or per anchor Classification (CE) + box regression (L1/smooth L1); matching (e.g. Hungarian) for set prediction (DETR). mAP (mean average precision) at IoU thresholds; COCO AP@.5, AP@.5:.95.

Choosing the right setup

  • Single label from fixed set → Multi-class: linear (or MLP) head with num_classes outputs, cross_entropy, accuracy.
  • Multiple non-exclusive labels → Multi-label: same head but BCE per class; macro F1 if classes are imbalanced.
  • Predict a number → Regression: linear head, MSE or MAE; eval with RMSE and R².
  • Generate text / sequence → Next-token CE; eval with perplexity and task-specific benchmarks.
  • Match or retrieve → Embeddings + contrastive loss; eval with Recall@k and NDCG.
  • Order matters (search, recommendation) → Ranking loss (pairwise or listwise); NDCG/MAP.
  • Agent / policy → RL loss (PPO, DPO, etc.); eval with return or downstream success.

Summary

Match the output shape and loss to the problem (classification → CE; regression → MSE; multi-label → BCE; sequence → CE with ignore_index). Then choose metrics that reflect your goal (accuracy, F1, NDCG, perplexity, mAP, etc.) and compute them on a held-out set or benchmark.
